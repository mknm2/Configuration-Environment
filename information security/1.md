# 🛡️ 项目开发安全规范：防范个人信息与敏感密钥泄露

在软件开发过程中，API Key、数据库密码、个人敏感信息（PII）或加密私钥等敏感数据的泄露，可能导致账号被盗刷、数据泄露甚至严重的安全事故。为确保代码仓库的安全，特制定本规范。

---

## 📌 一、核心防护原则

1. **代码与配置彻底分离** — 任何敏感信息严禁硬编码（Hardcode）在源代码中
2. **最小权限原则** — 生成的 API Key 或 Token 仅授予完成任务所需的最小权限，并设置合理的过期时间
3. **本地凭据隔离** — 开发环境的私密参数仅保存在本地，绝不跟随 Git 版本控制上传

---

## 🛠️ 二、配置文件与 Git 追踪

### 1. 优先建立 `.gitignore` 规则

在项目创建的第一时间（首次 `git commit` 前），必须配置好 `.gitignore` 文件：

```gitignore
# 环境变量与密钥文件
.env
.env.*
!.env.example
config.json
secrets.yml

# 身份凭证与私钥
*.pem
*.key
*.pfx
*.keystore

# IDE 及个人系统配置
.vscode/
.idea/
*.suo
*.user

# 打包与编译产物
build/
dist/
__pycache__/
*.exe
```

### 2. 使用环境变量与 `.env.example` 模板

- **本地真实密钥**：统一存放在 `.env` 中，代码通过环境变量（如 `os.getenv` / `process.env`）读取
- **团队共享模板**：提交仅含变量名、不含真实值的 `.env.example` 到 Git：

```bash
# .env.example（可安全提交到 Git）
DATABASE_URL="postgresql://user:password@localhost:5432/dbname"
OPENAI_API_KEY="your_openai_api_key_here"
SECRET_KEY="your_secret_key_here"
```

## ⚙️ 三、自动化安全拦截（CI/CD & Git Hooks）

人为注意难免疏漏，必须借助自动化工具实现"硬拦截"。

### 1. 本地 Pre-commit Hook

在 `git commit` 时自动扫描暂存区中的疑似密钥：

- **推荐工具**：[Gitleaks](https://github.com/gitleaks/gitleaks) 或 [pre-commit](https://pre-commit.com/)
- **生效机制**：一旦检测到匹配 `sk-proj-...`、`AWS_SECRET` 或 RSA 私钥特征，`git commit` 将被直接中断

### 2. 云端推送保护（Push Protection）

- GitHub / GitLab 仓库设置中开启 **Secret Scanning** 和 **Push Protection**
- 路径：*Settings → Security → Code security and analysis*
- 开启后，云端在接收 `git push` 时扫描并阻断含已知平台 Token 的推送

## 👨‍💻 四、编码与日志安全

### 1. 禁用硬编码"临时测试"

- 严禁为了调试将 API Key 直接写在代码中——临时硬编码极易因 `git add .` 而误提交

### 2. 日志与报错脱敏

- **控制台/日志脱敏**：打印日志时切勿直接输出密码、Token 或用户隐私（身份证、手机号、邮箱）
- **生产环境关闭 Debug**：Web 框架（Flask、Django、Express）上线时必须关闭 `DEBUG = True`，防止 Stack Trace 泄露服务器路径及环境变量

## 🚨 五、应急响应：泄露后的正确止损顺序

> ⚠️ 步骤顺序至关重要，请严格遵循。

### 步骤 1：立即废弃密钥（最高优先级）

- **切勿先研究怎么删 Commit！** 自动化爬虫抓取公开仓库只需几秒钟
- **立即**登录对应平台（OpenAI、阿里云、GitHub 等）**注销并重置该密钥**——密钥一旦失效，代码留在网上也是安全的

### 步骤 2：清理 Git 提交历史

**方案 A：仅有 1 个提交时（最简）**

```bash
git update-ref -d HEAD
git add -A
git commit -m "Initial commit"
git push -f origin main
```

**方案 B：多个提交时（孤立分支重建）**

```bash
git checkout --orphan temp-branch
git add -A
git commit -m "Initial commit"
git branch -D main
git branch -m main
git push -f origin main
```

### 步骤 3：清理 GitHub 缓存 / 重建仓库

- GitHub 会对 Git 对象保留缓存。对于极度敏感的泄露，最彻底的方案是**直接在 GitHub 上删除该仓库并重新创建**

---

## ✅ 六、提交前常见踩坑自查清单

每次 `git add` / `git commit` 前快速过一遍：

- [ ] 是否先执行了 `git status` 确认暂存内容？
- [ ] `.env` / `config.json` / `*.pem` 等是否已加入 `.gitignore`？
- [ ] IDE 配置目录（`.vscode/`、`.idea/`、`.claude/`）是否已忽略？
- [ ] 构建产物（`build/`、`dist/`、`__pycache__/`、`*.exe`）是否已忽略？
- [ ] 配置文件中是否包含绝对路径关键字？提交前搜索：`/home/`、`/Users/`、`C:\\`、`miniconda`、`AppData`
- [ ] 是否在 `.env.example` 中写了示例值而非真实密钥？
- [ ] 日志输出是否脱敏，没有打印 Token、密码、手机号等敏感信息？
